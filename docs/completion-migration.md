# Shell Completion Migration — Handoff

## TL;DR

We have hand-written fish/bash/zsh completion code appended to clap_complete's
static output. It works but is ~120 lines of shell glue we maintain by hand. We
want to replace it with **clap_complete's dynamic completion engine** (same
approach worktrunk uses) so all completion logic lives in Rust, attached to
clap args via `#[arg(add = ...)]`.

This document captures everything needed to execute that migration in a fresh
session.

---

## 1. Background — how we got here

User started by saying "the fish shell completions are kind of bad." That was
ambiguous — turned out to be a mix of:

- A stale `~/.config/fish/completions/wt.fish` file left over from worktrunk
  (calling `$WORKTRUNK_BIN` which still pointed to the old worktrunk binary →
  branch completions instead of jj workspaces). **User-side; resolved by
  deleting the file.**
- The user's `wt` fish function was an old version that only handled `switch`
  (no `cd:`/`exec:` directives, no `remove`). **User-side; resolved by
  re-running `jjwt config shell init fish`.**
- The user's `~/.config/jjwt/config.toml` overrides `worktree-path` to
  `{{ repo_path }}/.worktrees/{{ branch | sanitize }}`, which leaked into the
  e2e tests because the tests didn't isolate `HOME`/`XDG_CONFIG_HOME`. **Real
  test isolation bug; fixed (see §3).**
- Real shell-completion gaps in the crate itself. **Partially fixed (see §3),
  now we want to redo properly via dynamic completion.**

## 2. The two completion sources (current architecture)

There are two completion files for two different things:

- **`jjwt` completions** — generated by `clap_complete::generate(...)` from
  the clap argument tree in `src/cli.rs`. Knows every subcommand and flag.
  Blind to "what is a workspace name."
- **`wt` completions** — `wt` is a *fish function* (or bash/zsh function),
  defined by `jjwt config shell init <shell>`. The function exists because
  `jjwt switch` prints a path/directive that the parent shell needs to `cd`
  into, which a child process can't do. The function has its own completion
  rules.

These two sources didn't coordinate before our fixes. The CLI tree in
`src/cli.rs` is the natural source of truth; everything else should derive
from it.

## 3. What's already changed in this session

All changes are committed/staged-ready on the current branch.

### `src/cli.rs`
- `ConfigShellSub::Completions` handler now buffers clap's output, appends
  shell-specific suffix from `src/shell/cmd/shell.rs`, writes to stdout.
- Added hidden subcommands `step _complete-workspaces` (pre-existing) and
  `step _complete-bookmarks` (new) used by the hand-written shell glue.

### `src/shell/cmd/shell.rs`
- `FISH_COMPLETIONS` → `complete -c wt -w jjwt` (fish wraps `jjwt`).
- `BASH_COMPLETIONS` → `complete -F _jjwt_with_workspaces wt`.
- `ZSH_COMPLETIONS` → `compdef _jjwt_with_workspaces wt`.
- New `FISH_COMPLETIONS_SUFFIX`, `BASH_COMPLETIONS_SUFFIX`,
  `ZSH_COMPLETIONS_SUFFIX` constants holding the hand-written dynamic
  completion shell glue. These are what we want to **delete** in the
  migration.

### `src/shell/cmd/complete.rs`
- Added `run_bookmarks(cwd, prefix)` alongside the existing
  `run(cwd, prefix)` (workspaces). Both are called by the hidden subcommands
  above.

### `tests/e2e_smoke.rs`
- Added `jjwt_cmd(fake_home: &Path) -> Command` helper that sets `HOME` and
  `XDG_CONFIG_HOME` to a per-test temp dir.
- All five `Command::new(jjwt_bin())` sites in the three e2e tests now use
  this helper, so the developer's `~/.config/jjwt/config.toml` can't leak in.

### `README.md`
- Install order reversed: completions first, then `wt` wrapper.
- Fish wrapper install path changed from `~/.config/fish/functions/wt.fish`
  to `~/.config/fish/conf.d/wt.fish` (so the `complete -c wt -w jjwt` line
  inside actually gets sourced at startup).

### What current completions cover
- Workspaces on `switch`, `remove`, `step relocate` (1st positional),
  `step copy-ignored` (1st + 2nd positional).
- Bookmarks on `switch -b/--base`.
- Directories on `-C/--chdir`.
- Suppressed file fallback on `switch -x/--execute` and on `switch`/`remove`
  positionals past the last meaningful slot.

All three shells (fish, bash, zsh) work. Tests pass.

## 4. The plan — switch to dynamic completion (worktrunk's model)

### 4.1 What worktrunk does

Reference: `/tmp/worktrunk/src/completion.rs`, `/tmp/worktrunk/src/cli/*.rs`.

1. **`Cargo.toml`** uses `clap_complete` with its (unstable) dynamic engine:
   ```toml
   clap_complete = { version = "4", features = ["unstable-dynamic"] }
   ```

2. **`src/completion.rs`** defines a `maybe_handle_env_completion()`
   function:
   - Returns immediately unless `$COMPLETE` env var is set.
   - Walks `clap::Command` via `clap_complete::engine::complete()`, which
     calls `ValueCompleter::complete(current_token)` on whichever arg the
     cursor is on.
   - Emits candidates in the shell's expected format (zsh uses `:` between
     value and help, fish uses `\t`, bash has no help separator).

3. **`ValueCompleter` impls** for dynamic candidate sources:
   ```rust
   pub(crate) fn branch_value_completer() -> ArgValueCompleter { ... }
   pub(crate) fn worktree_only_completer() -> ArgValueCompleter { ... }
   pub(crate) fn alias_name_completer() -> ArgValueCompleter { ... }
   pub(crate) fn hook_command_name_completer() -> ArgValueCompleter { ... }
   ```
   Each is a struct implementing `clap_complete::engine::ValueCompleter` that
   returns `Vec<CompletionCandidate>`. They can use a thread-local
   `CONTEXT` to inspect the full command line — e.g. suppressing branch
   completion on `switch` when `--create` is also present.

4. **Clap args** are annotated:
   ```rust
   #[arg(add = crate::completion::worktree_branch_completer())]
   branch: String,
   #[arg(short = 'b', long, add = crate::completion::branch_value_completer())]
   base: Option<String>,
   ```
   No more hand-written shell.

5. **`main.rs`** calls the entry point before parsing:
   ```rust
   fn main() {
       if completion::maybe_handle_env_completion() {
           return;
       }
       // ... normal CLI dispatch
   }
   ```

6. **Shell registration** is a one-liner per shell, generated by
   `CompleteEnv`. For fish (from worktrunk's snapshot):
   ```fish
   complete --keep-order --exclusive --command wt --arguments \
     "(test -n \"$WORKTRUNK_BIN\"; or set -l WORKTRUNK_BIN (type -P wt 2>/dev/null); \
       and COMPLETE=fish $WORKTRUNK_BIN -- \
         (commandline --current-process --tokenize --cut-at-cursor) \
         (commandline --current-token))"
   ```
   `CompleteEnv::with_factory(cmd).try_complete([], None)` produces this
   automatically when invoked with `$COMPLETE` set to a shell name and no
   trailing tokens.

### 4.2 Migration steps for jjwt

1. **Cargo.toml**
   - Change `clap_complete = "4"` to
     `clap_complete = { version = "4", features = ["unstable-dynamic"] }`.

2. **Create `src/completion.rs`**
   - Port worktrunk's `maybe_handle_env_completion` as-is (it's mostly
     boilerplate around `clap_complete::engine::complete` + shell-aware
     output formatting).
   - Implement two `ValueCompleter` structs:
     - `WorkspaceCompleter` — uses `JjLib::new(cwd)` +
       `jj.workspace_list(repo_root)`. Reuses existing logic from
       `src/shell/cmd/complete.rs::run`.
     - `BookmarkCompleter` — same but `jj.bookmarks_local(repo_root)`.
   - Expose constructor fns:
     ```rust
     pub(crate) fn workspace_completer() -> ArgValueCompleter
     pub(crate) fn bookmark_completer() -> ArgValueCompleter
     ```
   - Optional context-aware variants:
     - `WorkspaceCompleterRestricted` that suppresses output when `--create`
       is present in the command line (mirrors worktrunk's
       `suppress_with_create`). Use a thread-local `CONTEXT` to inspect
       full argv during completion.

3. **`src/main.rs`**
   - At the very top of `main()`:
     ```rust
     if jjwt::completion::maybe_handle_env_completion() {
         return Ok(());
     }
     ```
     (Adjust path based on whether `completion` is in the lib or bin.)

4. **`src/cli.rs` — annotate args**
   - `SwitchCmd::name`: `add = crate::completion::workspace_completer()`
     (consider the restricted variant that suppresses on `--create`).
   - `RemoveCmd::names`: workspace completer.
   - `SwitchCmd::base`: bookmark completer.
   - `StepSub::Relocate::old_name`: workspace completer.
   - `StepSub::CopyIgnored::source` / `dest`: workspace completer.
   - Hook name positional (`HookCmd::name`): hook-name completer (reads
     project config). Optional; can defer.
   - Alias name positionals: alias-name completer. Optional; can defer.

5. **`src/shell/cmd/shell.rs` — slim down to wrapper + registration**
   - Keep `FISH_WRAPPER` and `POSIX_WRAPPER` as-is.
   - Delete `FISH_COMPLETIONS_SUFFIX`, `BASH_COMPLETIONS_SUFFIX`,
     `ZSH_COMPLETIONS_SUFFIX`.
   - Replace `FISH_COMPLETIONS`/`BASH_COMPLETIONS`/`ZSH_COMPLETIONS` with
     the per-shell `CompleteEnv` registration lines for **both** `jjwt` and
     `wt`. The `wt` registration calls the same `jjwt` binary; pass `wt` as
     the program name so clap routes through the same tree.
   - Alternatively, keep init output to just the wrapper function and tell
     users to install completions separately via the
     `config shell completions` command.

6. **`src/cli.rs` — Completions handler**
   - Replace the current buffer-clap-and-append logic with:
     ```rust
     ConfigShellSub::Completions(c) => {
       clap_complete::CompleteEnv::with_factory(|| Cli::command())
         .completer(...)
         .try_complete(std::env::args_os(), None)?;
       Ok(())
     }
     ```
     Or more directly, just print the registration line for the requested
     shell. CompleteEnv emits this when invoked with no args after `--`.

7. **Delete dead code**
   - `step _complete-workspaces` and `step _complete-bookmarks` subcommands
     in `src/cli.rs` (the `StepSub::CompleteWorkspaces` /
     `CompleteBookmarks` variants and their dispatch arms).
   - `src/shell/cmd/complete.rs` entirely — the logic moves into
     `WorkspaceCompleter`/`BookmarkCompleter` in `src/completion.rs`.

8. **`README.md` — update install steps**
   - One per-shell completion install command (e.g.
     `COMPLETE=fish jjwt > ~/.config/fish/completions/jjwt.fish`).
   - The `wt` wrapper install stays similar but its completion line is
     replaced by another `CompleteEnv` registration that points at `wt`.

9. **Tests**
   - `tests/e2e_smoke.rs` already isolates HOME/XDG; no changes needed.
   - Optional: snapshot tests over the registration lines emitted for each
     shell to lock the public contract.

## 5. Open questions / decisions for the new session

- **Restrict workspace completion on `--create`?** Worktrunk does this for
  `--create` (you're naming a *new* worktree, so existing branches shouldn't
  match). Same logic likely applies to `jjwt switch --create`. Recommend
  yes, mirroring worktrunk's `suppress_with_create`.
- **Bookmark filtering for `--base`?** Worktrunk has three branch
  completer flavors (`branch_value_completer`, `worktree_branch_completer`,
  `local_branches_completer`, `worktree_only_completer`). Decide whether
  `--base` should include remote tracking refs, only locals, etc.
- **Hook name completion?** `jjwt hook <name>` could list configured hook
  names from project + user config. Worktrunk does this. Optional for first
  pass.
- **Alias name completion?** `jjwt <alias>` external subcommands could be
  injected as stubs into the clap tree at completion time (worktrunk's
  `inject_alias_subcommands`). Defer.
- **Where does `completion.rs` live?** Bin-only (`src/bin/completion.rs`
  doesn't exist here) or under `src/`? Worktrunk has it at `src/completion.rs`
  alongside `main.rs`. Probably fine here too since jjwt is single-binary.
